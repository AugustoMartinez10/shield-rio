BACKEND IMPLEMENTADO
Archivo principal: `server.js`

Se implemento:
- `POST /api/contact`
- validacion de payload
- sanitizacion de campos
- honeypot anti-spam usando el campo oculto `website`
- rate limit basico en memoria por IP
- entrega por dos modos:
  - `log`: imprime la consulta en consola
  - `smtp`: envia mail real por SMTP

Validaciones actuales:
- `name` obligatorio
- `email` obligatorio y con validacion
- `phone` opcional, pero validado si llega
- `serviceInterest` opcional, validado contra lista permitida
- `message` obligatorio y con limite de longitud
- `privacyConsent` obligatorio
- `website` debe venir vacio para pasar honeypot

SERVICIO DE EMAIL
Proveedor elegido: Brevo

Motivo de eleccion:
- simple de configurar
- suficientemente prolijo para un formulario web
- gratis para arrancar

Configuracion usada:

- `SMTP_HOST=smtp-relay.brevo.com`
- `SMTP_PORT=587`
- `SMTP_SECURE=false`

Estado de Brevo:

HARDENING APLICADO EN BACKEND
Se aplicaron medidas basicas de bajo riesgo en `server.js`:
- `app.disable('x-powered-by')`
- header `Referrer-Policy: strict-origin-when-cross-origin`
- header `X-Content-Type-Options: nosniff`
- header `X-Frame-Options: DENY`
- header `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- header `Cross-Origin-Resource-Policy: same-origin`
- `Strict-Transport-Security` solo en produccion
- rechazo de requests a `/api/contact` que no lleguen como `application/json`
- limpieza basica del store del rate limit cuando no quedan timestamps vigentes

COSAS A NO OLVIDAR
- Si el servidor ya esta corriendo y se intenta levantar otro en el mismo puerto, aparece `EADDRINUSE`.
- Eso no implica que el proyecto este roto; solo indica que ya hay una instancia escuchando.
- El archivo `.env` esta ignorado por git, como corresponde.
- No subir credenciales reales al repositorio.
