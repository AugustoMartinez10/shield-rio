Session Notes - 2026-04-03
Proyecto: Shield Rio
Objetivo de la sesion: correcciones responsive mobile, con revision iterativa pantalla por pantalla.

Resumen general
Durante esta sesion se trabajo principalmente sobre la experiencia mobile del sitio, corrigiendo problemas de navegacion, jerarquia visual, overlays sobre video, espaciados afectados por header fijo y presentacion de bloques complejos como tablas o grids interactivos. En casi todos los casos se mantuvo la experiencia desktop y se adapto solo la version responsive.

Archivos trabajados
- index.html
- quem-somos.html
- a-blindagem.html
- servicios.html
- contato.html

1. Menu hamburguesa mobile desbordado horizontalmente
Problema
- En mobile, al abrir el menu hamburguesa, el panel podia quedar fuera del viewport por la derecha, especialmente en 320px.

Solucion aplicada
- Se ajusto el contenedor `.mobile-menu` para que actue como referencia relativa.
- Se redefinio `.mobile-menu-panel` para anclarse a la derecha del boton.
- Se limito el ancho real del panel con viewport:
  - `width: min(18rem, calc(100vw - 2rem))`
  - `max-width: calc(100vw - 2rem)`
- Se anadio `box-sizing: border-box`.

Archivos afectados
- index.html
- quem-somos.html
- a-blindagem.html
- servicios.html
- contato.html

2. Menu hamburguesa: cierre tocando fuera
Problema

3. Header fijo tapando contenido superior en mobile
Problema
- En varias views, el contenido superior quedaba oculto o demasiado pegado al header fijo.
- El topbar superior en mobile tiene altura variable por wrap, pero el hero usaba `padding-top` fijo.

Solucion aplicada
- Se midio la altura real del `header` con JS y se guardo en `--mobile-header-height`.
- Se cambio el `padding-top` de `.mobile-hero` a un calculo dinamico:
  - `padding-top: calc(var(--mobile-header-height, 9.75rem) + 1.25rem);`
- Se agrego `scroll-margin-top` para secciones ancladas:
  - `section[id]`
  - `article[id]`
  - `footer[id]`
- La medicion se actualiza con:
  - `load`
  - `resize`
  - `orientationchange`
  - `ResizeObserver`

Archivos afectados
- index.html
- quem-somos.html
- a-blindagem.html
- servicios.html
- contato.html

4. Clases Tailwind que seguian pisando el offset superior
Problema
- Aunque se agrego el calculo dinamico del header, varios heroes seguian teniendo `pt-36` o `pt-40` en el HTML, lo que generaba conflicto.

Solucion aplicada
- Se eliminaron utilidades `pt-36` y `pt-40` en los heroes mobile para que el offset dependiera solo del sistema dinamico.

7. Home (`index.html`) - seccion "Tecnologia & Qualidade"
Problema
- En mobile no se entendia la interaccion de tocar un item para cambiar imagen/texto porque la imagen quedaba fuera del foco visual.

Solucion aplicada
- En mobile se reemplazo la experiencia por un acordeon visual.
- Cada item muestra inline:
  - imagen
  - titulo
  - descripcion
  - tag tecnico
- Solo una card queda abierta a la vez.
- Mas adelante se ajusto para que una card abierta tambien pueda cerrarse al volver a tocarla.
- En desktop se mantuvo el layout original de 3 columnas.
- Se reutilizaron los datos existentes (`techData`).

Archivo afectado
- index.html

10. `a-blindagem.html` - aclarar visualmente hasta que nivel llega la cobertura
Problema
- En responsive no quedaba suficientemente claro que el nivel ofrecido es `III-A`, que niveles inferiores quedan implicita o logicamente cubiertos y que `III` queda aparte.

Solucion aplicada
- Se rehizo la jerarquia visual del bloque mobile sin explicarlo de forma literal.
- Se introdujo una progresion visual:
  - `I -> II-A -> II -> III-A`
- `III-A` quedo como culminacion visual de la escala.
- `Nivel III` paso a un bloque separado como `patamar seguinte`.
- Se renombro el encabezado mobile del bloque a `Padrao Shield Rio`.
- El acordeon secundario paso a leerse como referencia tecnica:
  - `Leitura Tecnica`
  - `Abrir referencia da norma`

Archivo afectado
- a-blindagem.html

Nota
- El usuario decidio dejar este bloque "asi por ahora".
- Puede requerir refinamiento visual futuro si se busca una lectura todavia mas inmediata.

15. `contato.html` - espacio sobrante debajo del mapa
Problema
- Aun con el mapa visible, quedaba un espacio vacio debajo.
- Primero parecio un problema de altura del `iframe`, pero no era eso.

Diagnostico real
- `.location-stack` es un grid.
- Dentro de la columna del layout, se estaba estirando y distribuyendo altura extra entre sus filas.
- Eso hacia crecer artificialmente el bloque del mapa.

Solucion aplicada
- En `.location-stack` se agrego:
  - `align-content: start;`
  - `grid-auto-rows: max-content;`
- Ademas se ajustaron alturas del mapa:
  - desktop: `360px`
  - mobile: `260px`

Resultado
- El usuario confirmo que el espacio sobrante quedo corregido.

Archivo afectado
- contato.html

Criterios acordados durante la sesion
- Las correcciones se priorizaron en mobile.
- Siempre que la experiencia desktop estuviera bien, se intento preservarla.
